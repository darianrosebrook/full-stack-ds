# Remote Figma MCP Skill Surface

Use this surface as the preferred Figma MCP path when available. It supports file/design inspection and, when the connected MCP client/server exposes write-to-canvas tools, native Figma canvas setup.

## Purpose

The remote Figma MCP is the default first choice for answering and acting on:

- Which file is the target library file?
- What pages, components, component sets, styles, variables, and libraries already exist?
- What naming conventions or collisions must the materializer respect?
- Can the agent create or update native Figma content directly through write-to-canvas tools?
- Is the file already published as a library, and what observable metadata is available?

## Preferred reads

When tools exist, prefer reads in this order:

1. Resolve file identity from URL or file key.
2. Read file metadata.
3. Read page list and top-level node inventory.
4. Read existing component/component-set names.
5. Read style and variable collections.
6. Read comments or review notes only when relevant to setup decisions.

## Writes

Remote MCP write-to-canvas tools are allowed when all are true:

- The user requested library setup or another bounded Figma mutation.
- The MCP exposes the required write operation.
- The agent has produced a bounded change plan naming the affected Figma objects.
- The operation does not delete or overwrite user-authored work without stable generated provenance.

Preferred write targets:

- Documentation pages and frames.
- Component pages.
- Native frames and auto layout.
- Components and component sets when the MCP exposes those operations.
- Variables and styles when the MCP exposes those operations.
- Node metadata/provenance when the MCP exposes a supported mechanism.

Publishing a library, changing permissions, moving files, deleting files, or editing team-level settings always requires explicit user authorization.

## Evidence to capture

For every remote MCP session, record:

- File key or URL.
- File name.
- Last modified timestamp when available.
- MCP surface used.
- Pages inspected.
- Existing component or component-set count when available.
- Collision candidates.
- Write operations attempted.
- Missing tool capabilities.

## Collision handling

If Figma state already contains names matching generated descriptors:

- Prefer stable generated provenance when available.
- Otherwise report collisions before mutation.
- Do not infer that a same-name component is generated by Full Stack DS unless provenance data exists.

## Desktop fallback

If remote MCP is unavailable or lacks required capabilities, the agent may use desktop MCP only when configured by the user/environment. Treat that as a fallback or special-case path, not as the default mutation surface.
