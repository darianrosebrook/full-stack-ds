# Current Implementation Snapshot

This is the routing document for current project state. Read it before older architecture notes, ADRs, and successor-work docs when deciding whether a surface is doctrine, partial machinery, or proven machinery.

Snapshot point: `main@5f197b0bc45b5e229337bb0c5862b88035f9b112` (`fix(e2e): skip runtime-rail screenshots under CI=true via test.skip`) for all rows except where noted. Partial update at `main@5f256688` (2026-07-21) verified three surfaces only: the DOM-property reflection bindings row, the RN row's corpus count, and the styling-realization audits row. All other rows remain verified at the older snapshot point; the intervening drift (recursive composition/`ComponentInstanceIR`, composer slot-projection, `textOverflow`, Kotlin/Swift native compile lanes, iconography) is real and deferred to a dedicated reconcile spec — do not read untouched rows as re-verified.

This document is not a replacement for the underlying architecture docs. It is an index of claim strength: what is implemented, what the evidence actually proves, what it does not prove, and which older docs should no longer be read as the only truth surface.

## Freshness rule

Use this order when answering "what is true now?":

1. Current implementation snapshot.
2. Code and workflow surfaces at the cited commit.
3. Implemented/proven docs with `verified_at_commit`.
4. Architecture and reference docs.
5. the machine-local successor ledger (`docs/internal/successor-work.md`, when present) only for remaining work, not as a statement that everything listed is unimplemented.

If an older doc says a surface is future work but this snapshot identifies an implemented surface, treat the older language as stale until that specific doc is reconciled.

## Stale-doc corrections this snapshot makes explicit

Several older docs are still useful, but their frontmatter/status or prose lags the code:

- the machine-local successor ledger (`docs/internal/successor-work.md`) must not be read as "all listed surfaces are unimplemented." Some items are now partial or promoted, especially target-pack registry, component evidence pages, usage examples, and runtime rail.
- README still contains hand-maintained component-count language. Treat loader discovery under `packages/ds-codegen/src/contracts-fs.ts` as authoritative until the README count is generated or removed.
- The five-framework claim is still true but narrower than the broader target story. The current executable family is Web DOM; target packs and Figma descriptors extend the projection model but do not yet prove external pack execution or non-web runtime semantics.
- Runtime rail evidence is newer than the original architecture docs. Any answer that treats runtime preview/fact testing as future work is stale.
- The 2026-07-21 update is row-scoped: it verified only the DOM-property reflection bindings row, the RN row's corpus count, and the styling-realization audits row. Every other row still reflects the 2026-06-11 verification, and known intervening work (recursive composition/`ComponentInstanceIR`, composer slot-projection, `textOverflow`, Kotlin/Swift native compile lanes, iconography) is absent from this ledger until a dedicated reconcile spec re-verifies row by row.

## Current bounded thesis

The project is a contract-governed design-system substrate with generated Web DOM framework realizations, target-pack admission scaffolding, Figma descriptor projection, token governance, generated-artifact admission, runtime fact assertions, and partially derived component evidence pages.

The claim boundary is bounded. It does not yet prove production adoption ergonomics, complete accessibility adequacy, visual quality, npm-distributed packages, live Figma library publication, executable external target packs, or substrate-neutral UI semantics. The non-web mobile arc has narrowed — but not closed — the substrate-neutrality question: React Native is now a default-rail target with a generated package, typed native token projection, native variant/state style realization, real package-type admission, focused host-render + surface tests, and a public-export settings example typecheck lane, while simulator/device runtime behavior, full surface parity, compound-part emission, part-scoped/boolean-modifier styling, and platform accessibility parity remain unproven.
