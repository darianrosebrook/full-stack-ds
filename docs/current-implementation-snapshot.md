# Current Implementation Snapshot

This is the routing document for current project state. Read it before older architecture notes, ADRs, and successor-work docs when deciding whether a surface is doctrine, partial machinery, or proven machinery.

Snapshot point: `main@c5cda2c9` (2026-07-21, full reconciliation under `DOC-SNAPSHOT-RECONCILE-02`). Every row was re-verified against live code at this commit by dedicated read-only verification passes: five drift lanes (recursive composition, composer slot-projection, textOverflow/RN IR-authority, native compile lanes + rail descriptors, iconography) plus a staleness audit of every pre-existing row. Known surfaces still without a row: the visual-fidelity/token-enrichment campaign (its outcomes surface indirectly through the styling-realization audits row's gap counts) — recording it row-grade requires its own verification pass.

This document is not a replacement for the underlying architecture docs. It is an index of claim strength: what is implemented, what the evidence actually proves, what it does not prove, and which older docs should no longer be read as the only truth surface.

## Freshness rule

Use this order when answering "what is true now?":

1. Current implementation snapshot.
2. Code and workflow surfaces at the cited commit.
3. Implemented/proven docs with `verified_at_commit`.
4. Architecture and reference docs.
5. the machine-local successor ledger (`docs/internal/successor-work.md`, when present) only for remaining work, not as a statement that everything listed is unimplemented.

If an older doc says a surface is future work but this snapshot identifies an implemented surface, treat the older language as stale until that specific doc is reconciled.

The project is a contract-governed design-system substrate with generated Web DOM framework realizations, target-pack admission scaffolding, Figma descriptor projection, token governance, generated-artifact admission, runtime fact assertions, and partially derived component evidence pages.
